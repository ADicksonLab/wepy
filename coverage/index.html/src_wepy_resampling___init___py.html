1"""Resampling functionality for weighted ensemble simulations. 

2 

3Role of Resampling in WE 

4------------------------ 

5 

6The defining feature of weighted ensemble simulations (as opposed to 

7simply running multiple independent simulations in parallel) is the 

8addition of a processing step that allows for communication of the 

9states of the parallel simulations. In this communication step we have 

10the opportunity to make decisions about the relative productivity of 

11each of them and attempt to bias the overall behavior of the ensemble 

12of simulations, while not explicitly modifying the governing laws 

13(e.g. equations of motion, force field, hamiltonian, etc.) of each 

14parallel dynamics simulation universe. 

15 

16A resampler embodies this communication. At a basic level it takes as 

17input a set of samples derived from dynamics, called walkers. In 

18addition to the states (the random variable) the walkers also have an 

19associated probability or weight. At the beginning of a WE simulation 

20each walker in the ensemble has a uniform probability. But the 

21probabilities are free to change due to the action of the resampler, 

22that is due to a resampling process. 

23 

24Resampling characterizes a potentially large number of algorithms that 

25may or may not be useful. In the original WE algorithm by Huber and 

26Kim (TODO add citation) this resampling is achieved through the 

27"cloning" and "merging" of walkers. 

28 

29Outline of the resampler sub-package 

30------------------------------------ 

31 

32The design of the wepy framework allows for any user to implement 

33their own resampler. 

34 

35The only strict requirement that the framework is that a resampler 

36must have a method "resample" that receives a list of walkers and 

37returns some other list of walkers along with optional records the 

38resampling process and any changes to the state of the resampler 

39itself (while these records aren't strictly necessary for a simulation 

40to run, they are paramount to actually recording data on the 

41simulations). 

42 

43However, useful resamplers share a lot of shared functionality. This 

44sub-package attempts to provide a sub-framework that provides this 

45functionality. 

46 

47Decisions 

48========= 

49 

50First is the definition of the "decisions" and the decision records. 

51 

52A resampling process given a set of weighted samples should produce a 

53number of other samples where the set of states in the output is a 

54subset of those in the input. 

55 

56For example a simple form of resampling is subsampling, where we 

57simply drop some of the samples from the collection. Similar to how 

58outliers are removed from a dataset. 

59 

60In the subsampling mechanism we might say that there are only two 

61decisions a resampler can make: DROP and KEEP. 

62 

63These could then be embodied in a Decision class 

64(e.g. SubsamplingDecision) which enumerates the possibilities and 

65provides methods for performing the operations on a collection of 

66walkers given a set of these decisions. 

67 

68Such a resampler wouldn't be very useful for WE since the simulation 

69would simply fade away as each walker was eventually eliminated. 

70 

71A more complex (and useful) example would be to add a possible 

72decision to split up a walker into two. In this there are now two 

73samples of the same random variable (state) but the total probability 

74of that state is the same since the weights are normalized and divided 

75in half for each. 

76 

77These decisions might be called: DROP, KEEP, SPLIT. Now we can replace 

78dropped samples so that the number of samples is kept the same. 

79 

80We could also allow the splitting of a walker into multiple walkers 

81instead of just two, in this case we replace SPLIT with CLONE. The 

82CLONE record now needs some additional information to parametrize it's 

83action. A record containing the decision value and the decision 

84parameters is called a 'decision record'. A CLONE decision record would 

85look like for example: 

86 

87(decision=CLONE, n_children=3) 

88 

89If we add data about which walker it applies to on which cycle and 

90which step of resampling it would be a 'resampling record'. 

91 

92 

93One issue with the use of a pure subsampling scheme is that we must 

94renormalize the weights of the walkers when any are dropped. In WE the 

95sample distribution is preserved at each step by allowing the 

96'merging' of walkers instead of just removing them. Merging involves 

97choosing a walker/sample to remove and adding that weight to another 

98walker, before discarding the sample value. 

99 

100This cloning-merging strategy is the primary Decision class used in 

101wepy and the choice of most resamplers. The full listing of decisions 

102is: 

103 

104- NOTHING 

105- CLONE 

106- SQUASH 

107- KEEP_MERGE 

108 

109Where NOTHING and CLONE are as above, SQUASH is the walker that will 

110be discarded and have it's weight donated, and KEEP_MERGE is a walker 

111that will be receiving weight from one or more squashed walkers. 

112 

113Resamplers 

114========== 

115 

116Abstractly a resampler in the context of wepy is an algorithm that 

117given a sample of walkers produces instructions on how to resample 

118them. The instruction set is defined and implemented by a decision 

119object. 

120 

121Concretely (and state above), the Resampler class is parametrized by a 

122Decision class and has a method 'resample' which receives a set of 

123walkers and returns a resampled collection of walkers. 

124 

125Resamplers in the sub-framework also produce resampling records in 

126roughly the format described above so that storage formats can record 

127the lineages (family trees) of walkers. 

128 

129Changes in the state of the resampler can also be recorded for 

130profiling and analysis purposes if it is desired, but is completely 

131specific to the resampling algorithm. 

132 

133 

134It is important to note that the choice of resampler is entirely 

135independent of: 

136 

137- the type of sampling used e.g. molecular dynamics, monte carlo 

138- the system being simulated e.g. proteins, polymers 

139 

140The type of sampling being used is encapsulated in the Runner class. 

141 

142For the second point we would like our resamplers to be as resuable as 

143possible between different systems. 

144 

145For example, wepy has implementations of both the WExplore and REVO 

146resampling algorithms which are broadly applicable and interchangable. 

147 

148For instance if you want to simulate ligand unbinding for a particular 

149protein-ligand system you can choose either WExplore or REVO. This 

150choice can be made after you have already determined a metric for 

151discerning novelty in you protein-ligand system, as long as the metric 

152is encapsulated in a Distance class. 

153 

154Distances 

155========= 

156 

157A distance metric (in the context of wepy) is simply a function that 

158returns a single value when given the states of two walkers. 

159 

160For such a function to be considered a proper distance metric extra 

161criterion are required but wepy does not enforce these in 

162implementation. 

163 

164A Distance class is simply a class with a single method 'distance'. To 

165allow for certain basic performance requirements, it is also required 

166that the 'image' and 'image_distance' methods be overriden. This 

167allows for the transformation (which is potentially compute intensive) 

168of a walker state to a reduced form called an image. The 

169'image_distance' method should produce the same results as the 

170'distance' method for the same walker states. 

171
